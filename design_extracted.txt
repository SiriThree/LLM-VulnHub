信息安全科技创新课程项目
项目设计报告
LLM-VulnHub
面向 AI 大模型应用的漏洞库管理与智能分析系统
1. 项目背景与意义
随着大语言模型在智能问答、Agent、代码生成、办公自动化、企业知识库、智能客服等场景中的广泛应用，围绕大模型应用的新型安全风险也逐渐增多。与传统 Web 漏洞或系统漏洞相比，大模型安全风险不仅存在于代码层和网络层，还可能出现在 Prompt、工具调用、RAG 检索、插件权限、模型输出以及上下文管理等环节。
例如， Prompt Injection 可能诱导模型忽略系统提示词， RAG 知识库投毒可能影响模型回答结
果，Agent 工具调用缺乏权限控制可能导致越权操作，插件供应链风险可能扩大攻击面，模型输出不安全也可能进一步诱发业务风险。这些问题具有来源分散、描述非结构化、分类体系尚不统一、人工分析成本高等特点。
传统漏洞库通常主要面向操作系统、Web 应用、中间件、网络设备等对象，能够记录漏洞编号、漏洞描述、影响版本和修复建议， 但对 AI 大模型应用中的新型安全问题支持不足。AI 安全漏洞往往与具体业务流程、Prompt 设计、工具权限、知识库召回和模型网关策略相关，仅依靠普通关键词检索难以快速定位问题。
因此，本项目拟设计并实现一个面向 AI 大模型应用场景的漏洞库管理与智能分析系统。系统利用大模型对漏洞文本进行结构化抽取、风险分类、评分分析和修复建议生成， 并结合向量检索技术实现基于漏洞库的智能问答， 帮助用户更高效地管理、检索和分析 AI 大模型相关安全漏洞。
2. 项目目标与范围
本项目目标是构建一个具备完整交互能力的 AI 大模型漏洞库管理系统。系统不仅支持基本的漏洞增删改查，还能够利用大模型完成漏洞信息抽取、风险评分、智能问答和报告生成。项目按照“基础可交付 + 智能分析 + 工程化扩展”的思路组织功能。
2.1 MVP 功能边界
为避免项目范围过大导致难以按时完成，第一版系统应优先保证主流程闭环。MVP 版本需要支持：漏洞 CRUD、手动/URL 导入、AI 结构化抽取、标准化入库、风险评分、语义检索/RAG 问答、漏洞详情展示和基础数据看板。为了符合课题要求，“ 实时动态搜集 AI 漏洞”被设计为 P1 核心功能：通过
Celery Beat 定时任务或手动触发采集任务， 从配置好的公开来源周期性抓取新增漏洞文本， 再由 AI 自动判断相关性、抽取字段、去重并入库。
3. 项目名称、定位与使用对象
3.1 项目名称
项目名称暂定为：LLM-VulnHub：面向 AI 大模型应用的漏洞库管理与智能分析系统。该名称能够同时体现项目的安全方向、大模型方向和漏洞情报管理属性。
3.2 项目定位
本项目不是攻击工具，也不是单纯的漏洞 CRUD 管理系统， 而是一个结合大模型能力的 AI 安全情报管理平台。系统的核心定位是：面向 AI 大模型应用安全场景，构建一个支持漏洞信息管理、AI 结构化抽取、风险分级、RAG 智能问答与自动报告生成的漏洞情报管理系统。

4. 需求分析
4.1 功能性需求
4.1.1 漏洞管理功能
系统应支持对漏洞信息进行基本管理， 包括新增漏洞记录、修改漏洞信息、删除漏洞记录、查看漏洞详情、按关键词搜索漏洞、按风险等级/漏洞类型/影响组件/标签进行筛选，以及按风险评分和创建时间进行排序。
4.1.2 AI 结构化抽取功能
用户可以输入一段原始漏洞文本，例如安全公告、博客内容、论文摘要或漏洞描述。系统调用大模型自动抽取结构化字段， 并将抽取结果返回给前端供用户确认。为降低大模型幻觉风险，系统要求大模型无法判断的信息填入unknown， 并允许用户在入库前人工修改。
4.1.3 AI 漏洞动态采集与标准化入库功能
为了满足课题中“利用 AI 技术实时动态搜集 AI 漏洞， 并按照固定格式上传到 AI 大模型漏洞系统平台” 的要求，系统需要设计独立的动态采集模块。该模块允许用户在后台配置漏洞信息源， 并通过定时任务周期性抓取新增漏洞文本。采集到的原始文本不会直接入库， 而是先经过 AI 相关性判断、文本清洗、结构化抽取、相似去重和风险评分， 最后按照系统固定字段格式写入漏洞库。
课程项目中可将“实时动态”实现为可配置周期的准实时采集：例如每 5 分钟、30 分钟或 1 小时执行一次采集任务。答辩演示时既可以手动触发采集，也可以展示后台任务自动发现新漏洞、抽取字段并入库的完整流程。
4.1.4 风险评分功能
系统应根据漏洞字段自动计算风险评分。风险评分范围为 0 到 100，分数越高表示风险越大。为了避免完全依赖大模型导致评分不稳定，本项目采用“规则评分 + 大模型解释”的策略：规则系统负责计算分数，大模型负责解释评分原因和修复优先级。
4.1.5 RAG 智能问答功能
系统应支持用户基于漏洞库进行自然语言问答。用户可以询问Prompt Injection 的防护方式、
RAG 知识库泄露风险、某个组件相关的高危漏洞等问题。系统先在漏洞库中召回相关记录， 再将召回结果作为上下文提交给大模型生成回答。
4.1.6 相似漏洞检测功能
当用户录入新漏洞时，系统可以自动检索已有漏洞， 判断是否存在相似记录。该功能可减少重复录入，提高漏洞库质量。实现上，新漏洞文本被转化为向量后，与已有漏洞向量进行相似度搜索， 返回  Top-K 相似漏洞供用户确认。
4.1.7 报告生成功能
系统支持根据漏洞详情生成一份结构化分析报告。报告内容包括漏洞名称、漏洞类型、风险等级、风险评分、影响组件、漏洞描述、攻击方式、影响范围、修复建议、AI 分析结论和参考资料。MVP 阶段可以先实现 Markdown/页面展示，后续扩展 PDF 或 Word 导出。

5. AI 漏洞实时动态搜集与标准化入库设计
本节专门说明系统如何实现“ 实时动态搜集 AI 漏洞， 并按照固定格式上传到 AI 大模型漏洞系统平台”。系统将该能力拆分为信息源管理、采集任务调度、 内容解析、AI 相关性判断、AI 结构化抽取、去重合并、标准化入库和审核发布八个环节。
5.1 信息源管理
系统提供信息源配置能力， 每个信息源记录名称、类型、URL、采集周期、启用状态和最近采集时间。课程实现阶段可以优先支持 RSS/网页 URL/本地 JSON 三类来源，后续可扩展到 GitHub API、
CVE/NVD API 或企业内部安全公告系统。
5.2 采集任务调度
动态采集通过 Celery + Redis 执行， Celery Beat 负责周期性触发任务，Worker 负责抓取、解析和调用 AI 工作流。系统同时提供“立即采集”按钮，方便答辩演示。采集任务的执行状态、开始时间、结束时间、成功数量、失败数量和错误信息都会写入任务表， 便于追踪和排错。
5.3 AI 相关性判断
公开信息源中并非所有内容都属于 AI 大模型漏洞， 因此系统需要在入库前判断候选文本是否与
AI/LLM/Agent/RAG/模型供应链等主题相关。该步骤由关键词规则和大模型共同完成：规则先进行快速过滤，大模型再对候选文本进行语义判断， 并输出置信度和判断理由。
5.4 标准化抽取与固定格式上传
通过相关性判断的候选文本进入结构化抽取流程。系统要求大模型只输出符合 Schema 的 JSON 字段，随后使用 Pydantic 进行格式校验。如果 JSON 不合法或关键字段缺失，系统会自动重试一次；仍失败时将记录标记为“待人工处理”。  校验通过后，后端调用统一入库服务写入漏洞表、分析记录表和向量 Chunk 表。
5.5 去重、合并与审核机制
动态采集容易产生重复内容，例如同一个漏洞可能同时出现在 GitHub Advisory、安全博客和漏洞库中。系统采用三层去重策略：首先根据 source_url 判断完全重复；其次根据标题、组件、漏洞类型做关键词匹配；最后使用 pgvector 进行相似度检索。如果相似度超过阈值，系统不直接新建漏洞， 而是把新来源作为参考链接合并到已有漏洞，或将其标记为疑似重复等待人工确认。
5.6 课程实现范围
为了保证项目可完成，实时动态采集分为可演示版和增强版。可演示版优先实现本地测试源、
RSS/网页 URL 导入、手动触发采集和定时任务；增强版再扩展 GitHub API、更多公开漏洞源和多源合并。这样既能满足课题要求， 又能控制实现难度。
6. 技术栈设计与选型依据
本项目采用现代 AI 应用工程技术栈， 强调前后端分离、结构化数据与向量数据统一管理、AI 工作流可追踪和部署可复现。
6.1 选用 PostgreSQL + pgvector 的原因
漏洞库既需要结构化查询，也需要语义检索。结构化查询包括按漏洞类型、风险等级、影响组件、标签、时间和分数筛选；语义检索包括根据用户自然语言问题或新漏洞描述召回相似漏洞。使用
PostgreSQL + pgvector 可以在同一个数据库中同时完成两类需求， 避免结构化数据和向量数据分散在不同系统中。
6.2 选用 LangGraph 的原因
本项目的 AI 流程不是简单的一次问答， 而是由文本预处理、结构化抽取、漏洞分类、风险评分、修复建议、向量生成、报告生成等多个步骤组成。LangGraph 可以将这些步骤显式建模为节点， 便于记录中间结果、定位失败原因、重试失败节点和扩展新的分析逻辑。
6.3 开发策略
技术栈可以保持工程化， 但开发顺序应保持稳妥。第一阶段先同步调用跑通后端、数据库和 AI 抽取；第二阶段接入 pgvector 完成 RAG；第三阶段再引入 Celery + Redis 进行异步化；第四阶段完善前端页面、看板和 Docker Compose。
7. 系统总体架构设计
系统采用前后端分离架构， 整体分为前端展示层、后端服务层、AI 工作流层、数据存储层和异步任务层。
8. 核心业务流程设计
8.1 AI 漏洞动态采集与标准化入库流程
该流程直接对应课题要求中的实时动态搜集与固定格式上传。系统不是简单保存网页文本， 而是将采集、AI 判断、结构化抽取、去重、评分和入库整合为可追踪的自动化链路。
8.2 漏洞录入与 AI 分析流程
该流程是系统的核心主线，也是答辩时最适合演示的流程。用户只需要粘贴一段原始漏洞文本，系统即可完成结构化抽取、风险分析、修复建议生成和入库。
8.3 RAG 问答流程
RAG 问答的关键在于回答必须基于漏洞库上下文。系统应在回答后展示参考漏洞记录，方便用户追溯答案来源， 降低大模型幻觉带来的可信度问题。
8.4 风险评分流程
风险评分不完全依赖大模型， 而是采用“规则评分 + 大模型解释”的方式。这样既能提高分数稳定性， 又能保留可读的分析理由。

19. 项目总结定位
本项目以 AI 大模型应用安全为背景， 将传统漏洞库管理、大模型文本分析、向量检索和前后端工程系统结合起来，构建了一个面向 AI 安全场景的漏洞情报管理平台。
最终可以在课程报告中概括为：本系统围绕 AI 大模型应用安全中的漏洞情报管理问题，设计并实现了一个集动态漏洞采集、AI 相关性判断、标准化入库、漏洞管理、风险评估、智能检索问答和报告生成于一体的安全分析平台。系统通过 LangGraph 编排大模型分析流程，使用 PostgreSQL 与 pgvector 统一管理结构化漏洞数据和语义向量数据， 并通过 Next.js 与 FastAPI 实现可交互的前后端系统，具有较好的实用性、扩展性和课程展示价值。
